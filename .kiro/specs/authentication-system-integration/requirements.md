# Requirements Document

## Introduction

Integration of biometric authentication with a simulated credential-based authentication system. The feature implements the full banking-grade flow: login with username/password, opt-in biometric enrollment (hardware-backed key pair via Secure Enclave/TEE), and subsequent biometric login using challenge→signature→verification. The backend is simulated locally via a FakeAuthBackend, but all security properties (anti-replay, key invalidation, rate limiting) are implemented as learning objectives. Navigation uses a conditional stack pattern (auth stack vs. app stack) based on session state.

## Glossary

- **App**: The React Native simulated banking application (rn-biometrics-auth)
- **AuthRepository**: Domain interface for credential-based authentication operations (login, logout)
- **BiometricKeysRepository**: Domain interface in domain/biometrics/repositories/ for cryptographic key operations (createKeys, createSignature, deleteKeys, biometricKeysExist)
- **BiometricEnrollmentRepository**: Domain interface in domain/auth/repositories/ bridging auth and biometrics features for enrollment and biometric login orchestration
- **FakeAuthBackend**: Local datasource simulating server-side behavior: user validation, challenge issuance, signature verification, token emission, and rate limiting
- **AuthSession**: Entity representing an active session with userId, token, issuedAt timestamp, and authentication method (PASSWORD or BIOMETRIC)
- **Credentials**: Value object containing username and password for credential-based login
- **SecureStorageDatasource**: Data layer component wrapping react-native-keychain for persisting session tokens and enrollment flags with mandatory security configuration
- **BiometricKeysDatasource**: Data layer component wrapping react-native-biometrics key operations (createKeys, createSignature, deleteKeys, biometricKeysExist)
- **Challenge**: A unique, time-limited, single-use nonce generated by FakeAuthBackend for biometric signature verification
- **Login_Screen**: Authentication screen with credential form and conditional biometric login button
- **Home_Screen**: Post-authentication main screen with session information
- **Settings_Section**: Section within Home_Screen for biometric toggle, session state, and public key visualization
- **Enrollment_Modal**: One-time modal presented after first credential login offering biometric enrollment
- **Navigation_Stack**: Conditional React Navigation native-stack: auth stack (Login_Screen) when no session, app stack (Home_Screen/Settings) when session exists
- **Hardware_Inspector**: Existing educational panel in the biometrics feature showing hardware information
- **Strong_Biometrics**: Class 3 biometric authentication (hardware-backed, high security); required for all cryptographic operations
- **Result_Pattern**: The Result<T, E> type used for explicit error handling without exceptions

## Requirements

### Requirement 1: Credential-Based Authentication

**User Story:** As a banking app user, I want to log in with username and password, so that I can establish a session and access the application.

#### Acceptance Criteria

1. WHEN the user submits valid credentials on Login_Screen, THE AuthRepository SHALL return a Result containing an AuthSession with method set to "PASSWORD".
2. WHEN the user submits invalid credentials on Login_Screen, THE AuthRepository SHALL return a Result with kind "err" containing an AppError with an authentication failure code.
3. THE AuthRepository SHALL delegate credential validation to the FakeAuthBackend without storing the password in any persistent storage.
4. WHEN a successful AuthSession is established, THE Navigation_Stack SHALL transition from the auth stack to the app stack displaying Home_Screen.

### Requirement 2: Biometric Enrollment

**User Story:** As a banking app user, I want to optionally enable biometric login after my first credential login, so that subsequent logins are faster and more convenient.

#### Acceptance Criteria

1. WHEN the user completes a credential login for the first time and biometric enrollment does not exist, THE App SHALL present the Enrollment_Modal offering biometric enrollment.
2. WHEN the user accepts enrollment in the Enrollment_Modal, THE BiometricKeysRepository SHALL generate a hardware-backed key pair using createKeys.
3. WHEN key pair generation succeeds, THE BiometricEnrollmentRepository SHALL register the public key with the FakeAuthBackend and persist the enrollment flag in SecureStorageDatasource.
4. WHEN the user declines enrollment in the Enrollment_Modal, THE App SHALL record the rejection permanently and never present the Enrollment_Modal again for that user.
5. WHILE the device biometric capability reports only weak biometrics (below Class 3), THE App SHALL reject enrollment with an explanatory message indicating Strong_Biometrics are required.
6. IF key pair generation fails during enrollment, THEN THE App SHALL display an error message and allow the user to retry or dismiss.

### Requirement 3: Biometric Login Flow

**User Story:** As a banking app user with enrolled biometrics, I want to log in using my biometric sensor, so that I can access the app without entering credentials each time.

#### Acceptance Criteria

1. WHILE biometric enrollment exists for the current user, THE Login_Screen SHALL display a biometric login button alongside the credential form.
2. WHEN the user triggers biometric login, THE FakeAuthBackend SHALL generate a unique Challenge with a maximum lifetime of 60 seconds.
3. WHEN a Challenge is issued, THE BiometricKeysRepository SHALL invoke createSignature with the Challenge to produce a cryptographic signature authorized by biometric match.
4. WHEN a valid signature is produced, THE FakeAuthBackend SHALL verify the signature against the stored public key and return an AuthSession with method set to "BIOMETRIC".
5. WHEN the FakeAuthBackend successfully verifies the signature, THE FakeAuthBackend SHALL mark the Challenge as consumed so that the same Challenge cannot be used again.
6. IF the signature verification fails, THEN THE FakeAuthBackend SHALL increment the failed verification count for the user and return an authentication error.
7. IF the user cancels the biometric prompt during signature creation, THEN THE App SHALL remain on Login_Screen without error severity and allow the user to retry or use credentials.

### Requirement 4: Anti-Replay and Challenge Security

**User Story:** As a security-conscious developer, I want the challenge-response flow to implement anti-replay protections, so that I understand server-side security properties for biometric authentication.

#### Acceptance Criteria

1. THE FakeAuthBackend SHALL generate each Challenge as a cryptographically random nonce unique per verification attempt.
2. WHEN a Challenge has been used for one signature verification (successful or failed), THE FakeAuthBackend SHALL reject any subsequent verification attempt using the same Challenge.
3. WHEN a Challenge has existed for more than 60 seconds without being used, THE FakeAuthBackend SHALL reject verification attempts with that Challenge as expired.
4. WHILE a user has exceeded the rate limit threshold for failed verification attempts, THE FakeAuthBackend SHALL reject further verification requests for that user with a rate-limit error.
5. THE FakeAuthBackend SHALL perform all challenge generation and signature verification server-side; the client SHALL never validate signatures locally.

### Requirement 5: Secure Storage Configuration

**User Story:** As a developer, I want all sensitive data stored with hardware-backed security configuration, so that session tokens and enrollment flags are protected by the device's security hardware.

#### Acceptance Criteria

1. THE SecureStorageDatasource SHALL configure Keychain accessibility as WHEN_PASSCODE_SET_THIS_DEVICE_ONLY preventing migration to backups or other devices.
2. THE SecureStorageDatasource SHALL configure access control as BIOMETRY_CURRENT_SET requiring a biometric match to release stored items and invalidating items when enrolled biometrics change.
3. WHILE the device operates on Android, THE BiometricKeysDatasource SHALL configure keys with setUserAuthenticationRequired set to true and restrict authentication to BIOMETRIC_STRONG (Class 3) only.
4. WHILE the device supports StrongBox (dedicated security chip), THE BiometricKeysDatasource SHALL prefer StrongBox-backed key storage with fallback to TEE when StrongBox is unavailable.
5. THE SecureStorageDatasource SHALL never store the user password in any form; only session tokens and enrollment flags are persisted.
6. THE App SHALL set allowDeviceCredentials to false for all biometric operations; device PIN or pattern SHALL NOT authorize cryptographic operations.

### Requirement 6: Session Management

**User Story:** As a banking app user, I want to manage my session explicitly, so that I can log out securely and understand session lifecycle.

#### Acceptance Criteria

1. WHEN the user triggers logout, THE AuthRepository SHALL clear the active AuthSession from SecureStorageDatasource.
2. WHEN the user triggers logout, THE App SHALL preserve biometric enrollment so the user can log in biometrically on the next session.
3. WHEN the AuthSession is cleared, THE Navigation_Stack SHALL transition from the app stack to the auth stack displaying Login_Screen.
4. WHEN the user triggers simulated session expiration from Settings_Section, THE App SHALL clear the AuthSession and transition to the auth stack as if the session expired naturally.
5. THE App SHALL persist AuthSession across app restarts using SecureStorageDatasource; WHEN the app launches with a valid persisted session, THE Navigation_Stack SHALL display the app stack directly.

### Requirement 7: Disable Biometrics

**User Story:** As a banking app user, I want to disable biometric login from Settings, so that I can revoke biometric access and revert to credential-only authentication.

#### Acceptance Criteria

1. WHEN the user disables biometrics from Settings_Section, THE BiometricKeysRepository SHALL invoke deleteKeys to remove the hardware-backed key pair.
2. WHEN biometric keys are deleted, THE BiometricEnrollmentRepository SHALL clear the enrollment flag from SecureStorageDatasource and deregister the public key from FakeAuthBackend.
3. WHEN biometric enrollment is removed, THE Login_Screen SHALL no longer display the biometric login button.
4. WHEN biometrics are disabled and the enrollment rejection flag was previously set, THE App SHALL clear the rejection flag allowing re-enrollment via Settings_Section in the future.

### Requirement 8: Key Invalidation on Biometric Change

**User Story:** As a banking app user, I want the app to detect when device biometrics change, so that unauthorized biometric enrollments on my device cannot access my account.

#### Acceptance Criteria

1. WHEN the user attempts biometric login and the signature operation fails due to key invalidation (biometric enrollment changed on device), THE App SHALL detect the invalidation error.
2. WHEN key invalidation is detected, THE BiometricEnrollmentRepository SHALL clear the enrollment state, delete invalidated keys, and deregister from FakeAuthBackend.
3. WHEN key invalidation is detected, THE App SHALL navigate to Login_Screen requiring credential-based re-authentication.
4. WHEN key invalidation is detected, THE App SHALL display an informational message explaining that device biometrics changed and re-enrollment is required.

### Requirement 9: Conditional Navigation

**User Story:** As a developer, I want navigation structured as conditional stacks based on session state, so that the app follows standard React Navigation patterns for authenticated flows.

#### Acceptance Criteria

1. WHILE no active AuthSession exists, THE Navigation_Stack SHALL render the auth stack containing Login_Screen as the only route.
2. WHILE an active AuthSession exists, THE Navigation_Stack SHALL render the app stack containing Home_Screen and Settings_Section.
3. THE Navigation_Stack SHALL use @react-navigation/native-stack as the navigator implementation.
4. WHEN session state changes (login or logout), THE Navigation_Stack SHALL switch stacks without manual navigation calls, relying on conditional rendering of the stack based on session state.

### Requirement 10: Educational Panel - Signature Flow

**User Story:** As a developer learning about biometric security, I want to see the challenge-response-verification flow visualized, so that I understand what happens at the hardware and protocol level.

#### Acceptance Criteria

1. THE Hardware_Inspector SHALL include a "Signature Flow" panel displaying the last biometric login attempt details.
2. WHEN a biometric login is attempted, THE Hardware_Inspector SHALL display the Challenge value issued by FakeAuthBackend.
3. WHEN a biometric login is attempted, THE Hardware_Inspector SHALL display the signature produced by createSignature (truncated for readability).
4. WHEN a biometric login is attempted, THE Hardware_Inspector SHALL display the verification result (success or failure with reason).
5. THE Settings_Section SHALL display the public key generated during enrollment for educational inspection.
6. THE Hardware_Inspector SHALL display the securityLevel of the device storage (SECURE_HARDWARE or SECURE_SOFTWARE) as reported by react-native-keychain.

### Requirement 11: Architecture and Layer Boundaries

**User Story:** As a developer, I want the authentication integration to respect clean architecture boundaries, so that features remain decoupled and testable.

#### Acceptance Criteria

1. THE App SHALL import react-native-keychain exclusively in data/auth/datasources/; no other layer or feature SHALL import react-native-keychain directly.
2. THE App SHALL import react-native-biometrics key operations (createKeys, createSignature, deleteKeys, biometricKeysExist) exclusively in data/biometrics/datasources/; no other layer or feature SHALL import these operations directly.
3. THE App SHALL not contain any import from data/biometrics/ within data/auth/ or from data/auth/ within data/biometrics/; cross-feature composition SHALL occur only through domain interfaces wired in the di/ layer.
4. THE BiometricKeysRepository interface SHALL reside in domain/biometrics/repositories/ as a separate interface from the existing BiometricRepository.
5. THE BiometricEnrollmentRepository interface SHALL reside in domain/auth/repositories/ as the consumer-side bridge between auth and biometrics features.

### Requirement 12: Unit Testing

**User Story:** As a developer, I want comprehensive unit tests for all authentication use cases and the FakeAuthBackend, so that security properties are verified without requiring a physical device.

#### Acceptance Criteria

1. THE FakeAuthBackend tests SHALL verify that a reused Challenge (replay attempt) is rejected with an appropriate error.
2. THE FakeAuthBackend tests SHALL verify that a Challenge older than 60 seconds is rejected as expired.
3. THE FakeAuthBackend tests SHALL verify that exceeding the rate limit threshold blocks further verification attempts for the affected user.
4. THE LoginWithCredentials use case tests SHALL verify successful login returns an AuthSession with method "PASSWORD" and failed login returns an AppError.
5. THE EnrollBiometrics use case tests SHALL verify that successful enrollment produces a public key and persists enrollment state.
6. THE LoginWithBiometrics use case tests SHALL verify the full challenge→signature→verification flow returns an AuthSession with method "BIOMETRIC".
7. THE DisableBiometrics use case tests SHALL verify that keys are deleted and enrollment state is cleared.
8. THE App tests SHALL use fake implementations of all repository interfaces without depending on device hardware or native modules.
